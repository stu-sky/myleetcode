# LeetCode 刷题项目 - AI 辅助约定

本项目为 LeetCode 刷题仓库，按 [Top 100 Liked](https://leetcode.cn/studyplan/top-100-liked/) 计划准备面试。AI 作为刷题辅助员，在用户提出无思路或思路一般的题目时，提供题解与代码。

## 技术栈

- Go 1.20+
- 代码与文档均使用 UTF-8 编码

## 文件结构

```
leetcode/
├── codes/          # 题解代码（Go）
├── docs/           # 题解文档（Markdown）
└── AGENTS.md       # 本文件
```

## 命名规范

统一格式：`{LeetCode 编号}-{英文题名}`

- 代码：`codes/0049-group-anagrams.go`
- 文档：`docs/0049-group-anagrams.md`

编号不足 4 位时前补 0（如 1 → 0001）。

## 代码约定

- **保持 LeetCode 原函数签名**，便于用户复制到 LeetCode 提交
- 只保留**一个**解法：选最易理解或最优解，不罗列多种

## 文档约定

- 开头需包含**题目**：题干、输入输出、示例，便于日后复习时无需查原题
- 内容包含：**思路**、**复杂度**、**易错点**、**变种思路**
- 文档内需包含跳转到对应 Go 文件的链接，例如：
  ```markdown
  **代码**：[codes/0049-group-anagrams.go](../codes/0049-group-anagrams.go)
  ```

### 题解叙述风格（重要）

在**简洁**的前提下，兼顾**可理解**与**可迁移**：

1. **知识点首次出现要「点一句」**  
   若正文里第一次用到**双端队列、前缀和、滑动窗口、单调栈/单调队列、哈希计数**等，用 **2～4 句话**说明：它是什么、在本题里扮演什么角色。不要求讲透体系，避免默认读者已经会。

2. **先写「怎么想到」，再写「怎么做」**  
   用一小节（如 **「怎么想到」** 或 **「思考路径」**）引导：**题目在问什么 → 朴素做法卡在哪（时间/空间/无法维护的信息）→ 为什么换成当前数据结构或技巧**。避免一上来只列步骤、不交代动机。

3. **控制总长**  
   介绍 + 思考路径 + 核心步骤 + 易错点，整篇思路部分以**一到两屏阅读**为宜；多用**短段落、列表**，少写冗长散文；与「点一句」和「引导思考」冲突时，**优先删重复、删次要细节**，不牺牲第 1、2 条。

### 变种思路一节写什么

- **必须包含同类 LeetCode 题**：列出与本题**核心套路相同或相近**的题目（带题号 + 简短题名），并**一句话**说明与本题的关系（如「同滑动窗口 + 字符计数」「定长窗口」）。方便按一条线刷题、复习。
- **可补充抽象变种**：如「改成本质不同但思路可对标的题型」，仍保持简短。
- **篇幅**：仍以表格或短列表为主，不展开成第二篇题解；没有特别合适的同类题时可不写满，不凑数。

## 行为边界

- 仅在用户明确请求某道题的题解时，才新增或修改 `codes/` 与 `docs/` 下的文件
- 不主动创建文档或代码，除非用户要求
- 回答问题时优先给出文字解释，再视需要补充代码或文档
