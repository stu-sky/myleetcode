# 构建 Claude Code 的经验：像 Agent 一样思考

> 原文：[Lessons from Building Claude Code: Seeing like an Agent](https://www.techtwitter.com/articles/lessons-from-building-claude-code-seeing-like-an-agent)  
> 作者：@trq212

---

## 引言

构建 Agent 框架时，最难的部分之一，是设计它的**动作空间**（action space）。

Claude 通过工具调用（Tool Calling）来行动，而在 Claude API 中，工具可以有多种形态：bash、skills，以及近年来的代码执行等（更多关于 Claude API 中程序化工具调用的内容，可参阅 @RLanceMartin 的新文章）。

面对这些选择，你该如何设计 Agent 的工具？是只需要一个像代码执行或 bash 这样的通用工具？还是为每种可能场景准备 50 个专用工具？

不妨把自己代入模型的视角：假设你拿到一道难题，你会希望手边有什么工具？答案取决于你自身的能力。

- **纸笔**是最低配，但你会受限于手算。
- **计算器**更好，但你需要会用它的高级功能。
- **电脑**最快、最强大，但前提是你知道如何写代码并运行。

这套思路同样适用于设计 Agent：你要给它**与其能力相匹配**的工具。那如何知道它有哪些能力？答案是：观察、阅读它的输出、反复试验。你要学会**像 Agent 一样思考**。

下面是我们从构建 Claude Code 的过程中总结出的几条经验。

---

## 一、改进「引导」能力：AskUserQuestion 工具

在开发 AskUserQuestion 工具时，我们的目标是提升 Claude 的**提问能力**（通常称为 elicitation，即「引导」）。

Claude 当然可以直接用纯文本提问，但我们发现，用户回答这些问题的体验并不好，感觉像是在浪费时间。如何降低这种摩擦，提高用户与 Claude 之间的沟通效率？

### 尝试一：修改 ExitPlanTool

我们首先想到的是在 ExitPlanTool 中增加一个参数，让它在输出计划的同时附带一组问题。实现起来最简单，但会让 Claude 困惑：它既要给出计划，又要给出关于计划的问题。如果用户的回答与计划冲突怎么办？Claude 是否需要调用两次 ExitPlanTool？我们需要换一种思路。

（关于为何设计 ExitPlanTool，可参阅我们关于 prompt caching 的文章。）

### 尝试二：改变输出格式

接着，我们尝试修改 Claude 的输出指令，让它用某种特定的 Markdown 格式来提问，例如输出带方括号备选项的 bullet 列表。我们再解析并渲染成用户界面。

这是最通用的改动，Claude 似乎也能按格式输出，但并不可靠。它可能会多加几句、漏掉选项，或干脆换一种格式。

### 尝试三：AskUserQuestion 工具

最终，我们决定单独做一个工具，Claude 可以在任意时刻调用，尤其在计划模式（plan mode）下会被提示优先使用。当工具被触发时，我们会弹出一个模态框展示问题，并阻塞 Agent 循环，直到用户回答。

这个工具让我们可以要求 Claude 输出结构化内容，并确保它给用户提供多个选项。用户也可以在 Agent SDK 中调用它，或在 skills 中引用，从而组合出更丰富的交互。

最重要的是，Claude 似乎很愿意调用这个工具，输出效果也不错。**再好的工具，如果 Claude 不知道如何调用，也等于没用。**

这是 Claude Code 中引导能力的终极形态吗？我们不确定。正如下一个例子所示，对某个模型有效的方式，未必适合另一个模型。

---

## 二、随能力演进：Tasks 与 Todos

Claude Code 刚推出时，我们发现模型需要一份 Todo 列表来保持专注。可以在开始时写好 Todos，随着工作推进逐项勾选。为此我们提供了 TodoWrite 工具，用于写入或更新 Todos 并展示给用户。

即便如此，我们仍经常看到 Claude 忘记该做什么。于是我们每 5 轮对话插入一次系统提醒，提醒它当前目标。

但随着模型能力提升，它不再需要这些提醒，反而觉得 Todo 列表成了束缚。频繁提醒 Todo 会让 Claude 认为必须严格按列表执行，而不是灵活调整。我们还发现 Opus 4.5 更擅长使用子 Agent，但子 Agent 如何共享同一份 Todo 列表？

基于这些观察，我们用 **Task 工具** 取代了 TodoWrite（更多关于 Tasks 的说明见此处）。Todos 侧重「让模型按计划推进」，Tasks 则侧重「让 Agent 之间相互协作」。Tasks 可以包含依赖关系，在子 Agent 之间共享更新，模型也可以修改或删除它们。

**随着模型能力增强，曾经必要的工具可能会变成限制。** 要不断重新审视「需要哪些工具」的假设。这也说明，只支持少数能力相近的模型，往往更易维护。

## 三、设计搜索界面

对 Claude 而言，**搜索类工具**尤为重要，它们能让它自行构建上下文。

Claude Code 最初使用 RAG 向量数据库为 Claude 检索上下文。RAG 强大且快速，但需要索引和配置，在不同环境中容易出问题。更重要的是，**上下文是「喂给」Claude 的，而不是它自己找的**。

既然 Claude 能在网上搜索，为什么不能搜索你的代码库？我们给了 Claude 一个 Grep 工具，让它自己搜索文件、构建上下文。

随着 Claude 变强，我们观察到一种模式：**只要给它合适的工具，它越来越擅长自己构建上下文**。

引入 Agent Skills 时，我们正式确立了**渐进式披露**（progressive disclosure）的思路：让 Agent 通过探索逐步发现相关上下文。

Claude 可以读取 skill 文件，这些文件又可以引用其他文件，模型可以递归读取。事实上，skills 的常见用法之一，就是为 Claude 增加搜索能力，例如教它如何调用 API 或查询数据库。

一年之内，Claude 从几乎无法自行构建上下文，进化到能在多层文件中进行嵌套搜索，精准找到所需信息。

**渐进式披露**如今已成为我们增加新功能而不新增工具时的常用手段。

## 四、渐进式披露实例：Claude Code Guide Agent

Claude Code 目前约有 20 个工具，我们一直在问自己：是否都需要？新增工具的门槛很高，因为每多一个工具，模型就多一个选项要权衡。

例如，我们发现 Claude 对「如何使用 Claude Code」了解不足。如果你问它如何添加 MCP，或某个斜杠命令的作用，它往往答不上来。

我们本可以把这些信息全部塞进系统提示，但用户很少问这类问题，那样会增加上下文负担，干扰 Claude Code 的主要任务：写代码。

于是我们尝试了一种渐进式披露：给 Claude 一个文档链接，让它按需加载和搜索。这能工作，但 Claude 会把大量检索结果塞进上下文，只为找到正确答案。

于是我们构建了 **Claude Code Guide 子 Agent**：当用户询问 Claude Code 自身时，会提示 Claude 调用它。这个子 Agent 有详细的指令，教它如何高效搜索文档、如何精简返回内容。

虽然还不完美——当用户问如何配置 Claude Code 时，Claude 仍可能困惑——但已经比之前好很多。**我们得以扩展 Claude 的动作空间，却没有新增工具。**

---

## 五、这是一门艺术，而非精确科学

如果你期待一套「如何构建工具」的硬性规则，很遗憾，本文给不了。**为模型设计工具，既是科学，更是艺术。** 它高度依赖你使用的模型、Agent 的目标，以及运行环境。

多实验、多读输出、多尝试新思路。**像 Agent 一样思考。**
